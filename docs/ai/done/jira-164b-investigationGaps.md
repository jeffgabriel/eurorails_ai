# JIRA-164b: Gap Analysis of Zero-Money Solvency & OnTrain Investigation

Audit of `docs/ai/jira-164-zeroMoneySolvencyAndOnTrainRoute.md` — verifying claims against actual code.

---

## Bug A: $0 Cash Death Spiral — Gaps

### 1. Broke-bot-gate threshold is wrong
- Doc says "$0 cash" but code checks `money < 5` (AIStrategyEngine.ts:373)
- Gate triggers at $0-$4, not just $0
- Proposed fix only handles `money === 0` — misses $1-$4 range

### 2. DiscardHand does NOT wipe the route
- `TurnExecutor.handleDiscardHand()` (lines 928-953) only replaces demand cards in DB
- Does NOT clear `activeRoute` or any route state
- The death spiral mechanism described in the doc is inaccurate

### 3. Missing: where does activeRoute get cleared?
- If DiscardHand doesn't clear the route, something else does
- Likely chain: discard -> demand cards change -> old route references stale demand cards -> route gets abandoned -> no new route at $0 -> PassTurn
- This chain is not traced in the doc

### 4. canDeliver scope mismatch in proposed fix
- Fix uses `context.canDeliver.find(d => d.isLoadOnTrain && d.isDeliveryOnNetwork)`
- But `canDeliver` is computed by `computeCanDeliver()` which only checks if bot is AT a delivery city
- `isLoadOnTrain` and `isDeliveryOnNetwork` are `DemandContext` properties on `context.demands`, not on `canDeliver` entries
- Fix should use `context.demands` instead

### 5. Proposed solvency recovery route is incomplete
- StrategicRoute type requires fields beyond what the fix shows: `upgradeOnRoute`, `estimatedTurns`, `tripPlanResponse`
- Missing optional fields may cause runtime errors in downstream consumers (TurnExecutorPlanner, route logging)

### 6. Phase 2 (borrowing) has no engine support
- Doc proposes implementing the borrowing mercy rule
- No `debt` field exists in the player/bot model
- No debt tracking, repayment, or net-worth calculation exists anywhere in the codebase
- This is a much larger change than implied — needs its own JIRA

### 7. Phase 3 guard may already be partially implemented
- Broke-bot-gate (line 380) already checks `noDeliverableOnNetwork` which evaluates `d.isLoadOnTrain && d.isDeliveryOnNetwork`
- If the bot IS carrying a load deliverable on-network, the gate should NOT fire
- Doc doesn't explain why it still fires — possibly because demand cards were already discarded in a prior turn, breaking the match

## Bug B: "OnTrain" Literal City — Gaps

### 8. "OnTrain" origin is misidentified
- Doc says it's in "demand card data"
- Actually a display string generated by `ContextBuilder.serializePrompt()` at line 1322: `const supplyNote = d.isLoadOnTrain ? 'OnTrain' : d.supplyCity`
- Root cause is **prompt leakage** — LLM sees "OnTrain" in prompt text and echoes it back as a city name in its route plan
- Not data corruption

### 9. Missing: RouteValidator gap
- If "OnTrain" leaks through as a route stop city, does RouteValidator catch and reject it?
- Not verified in the doc
- If RouteValidator accepts it, that's a second bug (missing validation)

### 10. Missing: LLM response parsing chain
- Doc jumps from "trip planner generates a pickup stop" to the fix
- Actual chain: LLM sees "OnTrain" in prompt -> LLM outputs it in structured response -> response parser creates RouteStop -> route executor uses it
- Response parser should also be considered as a fix location (reject "OnTrain" as a city)

### 11. skipCompletedStops may already handle this
- `isPickupComplete()` in routeHelpers.ts checks whether the train is carrying the load
- If carrying the load, the pickup IS already complete and should be auto-skipped
- Doc doesn't verify whether `skipCompletedStops` is actually being called for these routes — if it is, the bug may be that it's called too late or not at all in this code path

## Structural Gaps

### 12. No analysis of why the bot doesn't re-plan at $0
- The TripPlanner/LLM call path (line 369+) should still fire if the bot has an API key and no active route
- Doc doesn't explain why the LLM doesn't create a new delivery route when the bot has $0 but has carried loads and connected track
- The LLM prompt may not convey that movement on own track is free — bot may "think" it can't do anything

### 13. No analysis of noProgressTurns stuck detection interaction
- GuardrailEnforcer forces DiscardHand after 3+ no-progress turns (line 66-72)
- This could compound Bug A by repeatedly discarding hands that have valid delivery demand cards
- The stuck detection doesn't check whether the bot is carrying loads — same issue as Phase 3
